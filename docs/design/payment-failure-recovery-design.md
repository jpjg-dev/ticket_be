# 결제 장애 복구 설계 (보정 스케줄러)

## 문서 목적

PG 승인 이후 내부 상태가 반영되지 않는 장애를 어떻게 복구할지에 대한 설계 메모입니다. 현재 코드는 **PG 호출 실패(응답 불명)**까지는 처리하지만, **PG 성공 후 커밋 실패/크래시**는 처리하지 않습니다. 이 문서는 그 갭을 메우는 보정 스케줄러의 설계 근거를 정리합니다. (아직 미구현, 계획 단계)

## 문제 정의

결제 승인은 두 시스템에 걸쳐 있습니다.

```
PG(외부) 승인 = 돈 빠짐   ──┐
                            ├─ 둘은 하나의 트랜잭션으로 묶을 수 없음
내부 DB 상태 확정          ──┘
```

`PaymentService`는 `@Transactional`이라, `applyApproval()`은 메모리상 엔티티만 바꾸고 실제 커밋은 **메서드가 끝난 뒤 Spring 프록시**가 수행합니다. 그래서 다음 두 경우 불일치가 생깁니다.

| 실패 모드 | 설명 | 현재 처리 |
| --- | --- | --- |
| PG 호출 실패 (응답 불명) | 타임아웃·네트워크 끊김. 승인 여부 불확실 | ✅ `RestClientException` catch → 재조회 수렴 |
| PG 성공 + 커밋 실패 | 승인 직후 DB 커밋 실패 | ❌ 미처리 |
| PG 성공 + 프로세스 크래시 | 승인 직후 강제 종료(전원·OOM·kill) | ❌ 미처리 |

## 왜 인메모리 처리로는 부족한가

- 커밋 실패는 `applyApproval()` 이후, **메서드 바깥(프록시 커밋 시점)**에서 일어나므로 메서드 안의 어떤 체크로도 잡을 수 없습니다.
- 호출자 catch나 `afterCompletion` 훅으로 커밋 실패는 감지할 수 있지만, **프로세스 크래시는 실행 코드 자체가 사라져** 인메모리로는 절대 못 잡습니다.
- 따라서 크래시까지 견디려면 **살아남은 DB 상태를 나중에 다시 스캔**하는 방식만이 완전합니다.

## 방어 2계층

예약 만료를 수동 트리거 + 스케줄러로 처리한 것과 동일한 패턴을 적용합니다.

| 계층 | 역할 | 커버 | 한계 |
| --- | --- | --- | --- |
| 빠른 길 (인메모리) | 같은 요청 내 즉시 복구 | 호출 실패 모호함, 재시도 | 크래시 못 잡음 |
| 안전망 (보정 스케줄러) | 나중에 DB 스캔 후 수렴 | 커밋 실패·크래시·재시도 누락 | 즉시성 없음 |

빠른 길은 이미 구현됨(`RestClientException` catch + 재조회, 재시도 멱등). 추가할 것은 안전망(스케줄러) 하나.

## 보정 스케줄러 설계

### 1. 조회 키 — `orderId` (not `paymentKey`)

`paymentKey`는 `applyApproval()` 안에서 세팅되므로 **커밋 실패 시 함께 롤백**됩니다. 반면 `orderId`는 ready 시점에 저장되어 살아남습니다. 따라서 보정은 반드시 `orderId`로 PG를 조회합니다. 멱등키도 `"confirm:" + orderId`로 deterministic이라 롤백돼도 재생성 가능합니다.

### 2. 트리거 시점 — hold 만료 + 버퍼 (약 6분)

| 구간 | 담당 |
| --- | --- |
| 0~5분 (hold 윈도우) | 빠른 길 (재시도·재조회로 수렴 가능) |
| 5분 (hold 만료) | 만료 스케줄러가 `PENDING -> FAILED` 전이 |
| 6분~ | 보정 스케줄러가 집어감 |

- hold 만료(5분) **전에는 보정하지 않습니다.** 빠른 길이 수렴시킬 수 있어 침범하면 안 됩니다.
- 6분 = hold 만료(5분) + 만료 스케줄러 주기(1분) 버퍼. 만료 전이가 끝난 깨끗한 상태에서 보정합니다.

### 3. 락은 PG 조회 밖에서

```
[락 없음] orderId로 PG 상태 조회        ← 외부 호출, 락 안 쥠
[락]      상태 재확인 → 적용/취소        ← 짧게만 락
```

PG 조회를 락 안에서 하면 만료 스케줄러와 길게 경합합니다. 경합은 주기 조정이 아니라 **락 범위 축소**로 해결합니다. (주기로 race를 피하는 것은 신뢰할 수 없음.)

### 4. 복구 정책 — 만료 시 환불(cancel)

```
보정 대상(6분 경과 + 미확정) 각각:
  lookup = PG.조회(orderId)
  if lookup.status == DONE:        // 돈이 실제로 빠짐
      if 예약 유효(미만료·좌석 HELD):
          applyApproval → 예매 확정으로 수렴
      else:                         // 만료로 좌석 풀림(재판매 가능)
          PG.cancel(...) → 환불(보상)
  else:                             // 실제 미결제
      그대로 둠 (정상 만료 흐름)
```

만료로 좌석이 풀린 경우 무리하게 재확정하지 않고 환불합니다(좌석이 이미 팔렸을 수 있음).

### 5. 안전 윈도우 — 멱등키 15일

Toss 멱등키는 15일 유효합니다. 보정의 재조회/재호출은 이 윈도우 안에서 항상 안전합니다(같은 키 → 같은 결과). 단 15일은 "안전 상한"일 뿐 트리거가 아닙니다. 트리거는 hold 만료(6분)입니다.

## 정합성 안전 근거

- 만료 스케줄러는 `PENDING`을, 보정 스케줄러는 `FAILED`를 다뤄 생애주기가 달라 같은 row를 거의 동시에 건드리지 않습니다.
- 겹쳐도 락 순서 일관(Payment 먼저) + 상태 재확인으로 데드락·중복 처리가 발생하지 않습니다.

## 미적용 이유

- 결제 정합성의 핵심 경로라 동시성 테스트(만료-보정 경합) 없이 손대면 위험합니다.
- 현재 트래픽 규모에서는 빠른 길(재조회·재시도)로 대부분 수렴되며, 크래시 빈도가 낮아 우선순위를 뒤로 두었습니다.
- 구현 시에는 TDD로 동시성 테스트를 먼저 작성합니다.
