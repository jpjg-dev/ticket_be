# 인기 공연 전체 여정 성능 개선 요약

## 문서 목적

이 문서는 인기 공연 오픈 직후 사용자가 공연 목록, 상세, 좌석 조회, 예약 생성, 결제 완료까지 진행하는 전체 여정을 기준으로 성능 개선 과정을 정리합니다.

목표는 최대 처리량을 과장하는 것이 아니라, 같은 로컬 통제 조건에서 병목 가설을 하나씩 확인하고 예약/결제 정합성을 깨지 않는 범위에서 개선하는 것입니다.

## 기준 시나리오

| 항목 | 값 |
| --- | --- |
| 대상 이벤트 | `PERF_LOAD_TEST_EVENT` |
| 대상 회차 | `scheduleId=18` |
| 부하 테스트 좌석 | `391~1390` |
| 좌석 수 | `1,000` |
| 사용자 풀 | 성능 테스트 전용 사용자 `10,000`명과 Access Token Cookie |
| PG | 로컬 Mock PG |
| 유입 모델 | `ramping-arrival-rate` |
| 단계 | `10 -> 100 -> 300 -> 500 -> 1000 -> 100 journeys/s` |
| 실행 시간 | `50s` |

전체 여정 TPS는 `완료 iteration / 50s`로 계산했습니다. 결제 TPS는 k6의 `arrival_e2e_payment_completed` rate를 사용했습니다.

## 1차 개선: 공연 목록/상세 캐시

공연 목록과 상세는 카탈로그 성격이 강하고 자주 바뀌지 않으므로 `Spring Cache + Caffeine` 로컬 캐시를 적용했습니다.

선행 정리:

- 백엔드 `EventResponse`에서 시간 파생값인 `displayStatus`를 제거했습니다.
- 백엔드는 `bookingOpenAt`, `runStartAt`, `runEndAt` 원천 시간만 제공합니다.
- 프론트 서버 컴포넌트가 표시 상태를 계산합니다.
- 예매 오픈 전 예약 생성은 백엔드 `bookingOpenAt` 검증으로 차단합니다.

트레이드오프:

| 선택 | 이득 | 비용 / 보류 |
| --- | --- | --- |
| Caffeine local cache | Redis 없이 반복 조회 DB 접근을 줄일 수 있습니다. | 서버 재시작 시 cold cache가 생기고 다중 서버 공유는 되지 않습니다. |
| Redis 보류 | 현재 마무리 단계에서 중대규모 리팩토링을 피했습니다. | 분산 캐시/락/운영 복잡도는 후속 과제로 남습니다. |
| 좌석/예약 상태 캐시 제외 | 정합성 리스크를 줄였습니다. | 좌석 상태 조회 비용은 별도 개선이 필요합니다. |

## 2차 개선: 좌석 조회 트랜잭션 범위 분리

좌석 조회는 회차별 만료 처리를 포함합니다. 수동 만료를 제거하면 처리량은 좋아질 수 있지만, 스케줄러 주기 사이에 만료된 좌석이 판매 불가 상태로 남을 수 있습니다.

그래서 정책은 유지하고 트랜잭션 범위만 줄였습니다.

- `EventService.getSeats()`는 직접 write transaction을 감싸지 않습니다.
- 만료 상태 전이는 `ReservationExpirationService.expireByScheduleId()`의 write transaction에서만 처리합니다.
- 좌석 응답 매핑은 path variable의 `scheduleId`를 직접 사용해 `Seat -> Schedule` 접근을 줄였습니다.

트레이드오프:

| 후보 | 판단 |
| --- | --- |
| 수동 만료 제거 | 즉시 재판매 정책과 맞지 않아 보류했습니다. |
| 스케줄러만 의존 | 만료 후 재판매 지연 가능성이 있어 보류했습니다. |
| 트랜잭션 범위 분리 | 정책을 유지하면서 조회 경로 부담을 줄일 수 있어 적용했습니다. |

## 3차 개선: 카탈로그/예약 상태 분리

2차 개선 후에도 매진 여부가 좌석 조회 경로에서만 확정되어, 매진 이후에도 같은 조회 흐름이 반복됐습니다.

핵심 변경은 좌석 응답 크기 자체가 아니라 **공연 카탈로그 데이터와 실시간 예약 상태를 분리한 것**입니다. 좌석 목록을 내려주지 않는 처리는 이 설계의 보조 효과입니다.

적용 내용:

- `GET /api/v1/event/schedules/availability?scheduleIds=...` batch API를 추가했습니다.
- 좌석 조회 응답을 `{ scheduleId, soldOut, seats }` wrapper로 변경했습니다.
- `/seats`는 만료 처리 후 availability를 먼저 확인하고, 매진이면 좌석 목록 projection 조회를 생략합니다.
- 프론트 좌석 화면도 진입 회차 또는 선택 회차가 이미 매진이면 불필요한 좌석 조회를 생략합니다.

매진 기준:

```text
AVAILABLE = 0
HELD = 0
BOOKED > 0
```

`HELD`는 만료 후 `AVAILABLE`로 복구될 수 있으므로 매진으로 보지 않습니다.

트레이드오프:

| 후보 | 판단 |
| --- | --- |
| 좌석 상태 캐시 | 예약/결제/만료와 동기화 책임이 커서 제외했습니다. |
| event cache evict | 매진 감지마다 카탈로그 캐시 miss가 몰릴 수 있어 제외했습니다. |
| availability 별도 API | 카탈로그 캐시와 예약 상태를 분리할 수 있어 적용했습니다. |
| batch availability API | 여러 회차를 한 번에 확인해 메인/상세 서버 렌더링에 적합했습니다. |

## TPS 개선율

`전체 여정 TPS`는 결제 성공, 정상 경합 거부, 매진 거부까지 포함한 완료 여정 처리량입니다. `결제 TPS`는 실제 결제 완료만 집계하므로 1,000석 매진 이후에는 좌석 수에 의해 상한이 생깁니다.

| 단계 | 완료 iteration | 전체 여정 TPS | 초기 대비 | 직전 대비 | 결제 TPS | 결제 TPS 초기 대비 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Initial | `11,153` | `223.06 journeys/s` | - | - | `17.13 payments/s` | - |
| After event cache | `15,573` | `311.46 journeys/s` | `+39.63%` | `+39.63%` | `19.43 payments/s` | `+13.43%` |
| After seat transaction split | `15,739` | `314.78 journeys/s` | `+41.12%` | `+1.07%` | `19.64 payments/s` | `+14.65%` |
| After availability split | `16,847` | `336.94 journeys/s` | `+51.05%` | `+7.04%` | `19.92 payments/s` | `+16.29%` |

## 최종 판단

현재 가장 의미 있는 개선은 공연 카탈로그 데이터와 예약 상태를 분리해, 매진 이후에도 화면과 조회 흐름이 같은 좌석 조회를 반복하지 않게 만든 점입니다. 좌석 정합성 로직은 캐시하지 않고 DB 상태를 원천으로 유지했습니다.

이번 spike는 운영 최대 처리량 보장이 아니라 로컬 통제 환경의 포화 관찰과 개선 전후 비교입니다. `1000 journeys/s` 조건은 일반 운영 부하라기보다 매진 이후 반복 조회가 시스템을 무너뜨리는지 확인하기 위한 고부하 조건으로 해석합니다.

확정한 원칙:

- 대기열은 추가하지 않습니다. 이 프로젝트의 핵심은 대기열 노출이 아니라 좌석/예약/결제 정합성입니다.
- 좌석 상태는 캐시하지 않습니다.
- 공연 목록/상세는 카탈로그 캐시로 유지합니다.
- 예약 상태(`soldOut`, `available`, `held`, `booked`)는 카탈로그 캐시에 섞지 않습니다.
- 프론트 서버가 카탈로그 응답과 availability 응답을 조합해 화면 상태를 만듭니다.
- 예약 생성 API는 최종 정합성 방어선으로 계속 검증합니다.

## 후속 후보

- 좌석 상태 집계 쿼리의 `EXPLAIN` 확인
- 필요 시 `seats(schedule_id, status)` 인덱스 후보 실험
- 인덱스 적용 시 좌석 상태 변경 write 지표도 함께 비교
