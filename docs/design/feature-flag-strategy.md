# 대기열 Feature Flag 설계

## 목적

대기열을 한 번에 강제 적용하지 않고 `OFF -> SHADOW -> ENFORCED` 순서로 관측하며 전환하기 위한 운영 제어 기반입니다.

관리자 제어면과 실제 예약 진입 경로를 연결했습니다. `ENFORCED`에서는 Redis 대기열 입장과 토큰 검증을 통과해야 예약을 생성할 수 있습니다.

## 모드

| 모드 | 의미 |
| --- | --- |
| `OFF` | 기존 예약 흐름을 유지합니다. |
| `SHADOW` | 현재 대기 인원을 기준으로 판단 결과만 기록하고 예약은 기존 흐름으로 통과시킵니다. |
| `ENFORCED` | Redis 대기열에서 입장 허용 토큰을 받은 요청만 예약 생성 경계로 통과시킵니다. |

## 실행 흐름

```text
대기 등록 -> 회차별 Redis ZSET에 토큰 저장
순번 갱신 -> SSE로 ZRANK 기반 순번 전달
입장 허용 -> 스케줄러가 회차별 batch-size만큼 토큰 승격
예약 생성 -> 사용자/회차가 일치하는 입장 토큰을 원자적으로 claim
성공       -> 토큰과 대기 상태 제거
실패       -> 입장 토큰을 복구해 제한 시간 안에 재시도 허용
취소       -> claim 전인 본인 대기 항목만 원자적으로 제거
```

- 기본 배치는 3초마다 회차별 20명입니다.
- 대기 등록은 15분, 입장 허용은 2분, 예약 처리 중 claim은 30초 TTL을 사용합니다.
- 프론트는 `WAITING`이면 SSE 화면을 열고, `ADMITTED`를 받으면 선택한 좌석으로 예약 요청을 자동 재개합니다.

## 저장과 조회

- PostgreSQL을 영속 원본으로 사용합니다.
- 런타임 조회는 Redis의 `ticketledger:feature-flag:queue-mode`를 사용합니다.
- 캐시 값은 `version:mode` 스냅샷이며 TTL은 5초입니다.
- 캐시 미스일 때 DB를 조회하고 Redis Lua CAS로 더 최신 버전을 덮어쓰지 않도록 합니다.
- Redis 조회 또는 갱신이 실패하면 런타임 모드는 `ENFORCED`로 닫습니다. 대기열 상태를 확인할 수 없는 요청이 예약 경계를 우회하지 못하게 하고, Redis 접근 실패는 `503`으로 재시도하도록 합니다.

## 변경 정합성

- 관리자 변경 요청은 현재 `version`을 `expectedVersion`으로 보냅니다.
- `UPDATE ... WHERE id = 1 AND version = :expectedVersion` 조건부 업데이트의 영향 행 수로 충돌을 판정해, 오래된 관리자 화면의 덮어쓰기를 `409 Conflict`로 거부합니다.
- 단일 운영 설정값의 원자적 교체가 목적이므로 일반 엔티티 변경 감지용 `@Version`보다 명시적인 DB CAS를 선택했습니다.
- DB 커밋 후 Redis를 갱신합니다. Redis 갱신 실패 시 읽기 회로를 열어 stale 모드 대신 `ENFORCED`를 사용하고, 캐시 TTL로 잔존 시간을 제한합니다.

## 보안 경계

- API: `GET /api/v1/admin/feature-flags`, `PUT /api/v1/admin/feature-flags/queue-mode`
- `/api/v1/admin/**`는 기존 Spring Security의 `ROLE_ADMIN` 규칙으로 보호합니다.
- 컨트롤러와 서비스에서 관리자 정보를 다시 조회하거나 중복 권한 검사를 하지 않습니다.

## 자동 활성화 정책

운영자가 `SHADOW`를 선택하면 애플리케이션이 1초마다 대기열 진입 경계의 부하를 관찰합니다. `OFF`와 수동 `ENFORCED`는 운영자 override이므로 자동 제어가 개입하지 않습니다.

| 구분 | 요청률 | 예약 동시 실행 | Process CPU | Tomcat busy | Hikari pending |
| --- | ---: | ---: | ---: | ---: | ---: |
| 활성화 | `15 req/s` 이상 | `8` 이상 | `70%` 이상 | `70%` 이상 | `1` 이상 |
| 해제 | `6 req/s` 이하 | `3` 이하 | `50%` 이하 | `40%` 이하 | `0` |

- 활성화 조건 중 하나가 3회 연속 충족되면 `ENFORCED`로 전환합니다.
- 해제 조건은 모든 지표가 10회 연속 충족되고 Redis 대기 인원이 `0`일 때만 인정합니다.
- 전환 후 최소 30초 유지와 30초 cooldown을 적용해 임계값 부근에서 반복 전환되는 현상을 막습니다.
- 자동 제어가 활성화된 프로세스는 재기동 직후 `ENFORCED`로 시작합니다. 필수 메트릭이 모두 수집되고 해제 조건이 연속 충족된 뒤에만 `SHADOW`로 완화합니다.
- CPU, Tomcat, Hikari 중 필수 메트릭이 누락되거나 유효하지 않으면 복구로 판단하지 않으며, `SHADOW` 상태에서도 연속 누락은 과부하 신호로 처리합니다.
- 자동 활성화 평가는 전용 scheduler thread에서 실행해 결제 보정이나 예매 만료 작업이 길어져도 판단 주기가 밀리지 않게 합니다.
- 활성화 임계값은 항상 해제 임계값보다 크거나 같아야 하며, 잘못된 설정은 애플리케이션 시작 시 거부합니다.
- 자동 전환 상태는 프로세스 메모리에만 보관합니다. 현재 단일 인스턴스에는 충분하지만, 다중 인스턴스 전환 시에는 leader 선출 또는 공유 상태 저장이 필요합니다.
- DB CPU는 애플리케이션 내부의 즉시 판단 신호가 아니라 Prometheus 사후 검증 지표로 사용합니다. DB exporter 조회 장애가 입장 제어까지 전파되는 결합을 피하기 위해서입니다.

`SHADOW -> ENFORCED` 전환 직전에 이미 통과한 사용자가 예약 단계에서 토큰 없이 거부되지 않도록, 자동 제어가 켜진 `SHADOW`에서는 사용자·회차에 귀속된 일회성 bypass permit을 발급합니다. permit은 입장 토큰과 같은 소유권 검증 및 TTL을 사용하고 한 번만 claim할 수 있습니다.

노출 지표:

- `queue_auto_enforced`: 자동 강제 적용 여부 (`0` 또는 `1`)
- `queue_auto_activation_transitions_total`: 자동 전환 횟수
- `queue_arrivals_total`: 대기열 진입 경계 도착 요청 수
- `queue_protected_requests_active`: 예약 보호 구간의 현재 실행 수

## 장애 정책과 검증 결과

| 상황 | 정책 | 검증 결과 |
| --- | --- | --- |
| Redis 중단 | 대기열을 우회하지 않고 `503 Service Unavailable`과 `Retry-After: 1`을 반환합니다. | 실제 Redis 중단 상태에서 `3/3` 요청이 정책대로 거부됐습니다. |
| SSE 단절 | 연결 작업을 취소하고 같은 토큰으로 다시 연결할 수 있게 합니다. | 연결 자원 회수 후 동일 토큰 재연결에서 `ADMITTED`까지 이어지는 것을 확인했습니다. |
| 입장 토큰 만료·중복 사용 | 소유자, 입장 허용 상태, claim 여부를 Lua로 원자 검증해 예약 생성을 거부합니다. | 만료 상태와 중복 claim을 각각 거부했습니다. |
| 애플리케이션 재기동 | 대기 순번은 애플리케이션 메모리가 아니라 Redis ZSET을 원본으로 사용합니다. | store 인스턴스를 재생성한 뒤에도 기존 backlog 조회를 확인했습니다. |
| Redis 연결 복구 | 장애 중에는 `503`으로 닫고 Redis가 복구되면 기존 토큰을 다시 조회합니다. | Redis 연결 중단·복구 후 같은 토큰과 순번을 이어서 조회했습니다. |
| Redis 재기동 | AOF `everysec`로 내구성 구간이 지난 대기 상태를 복구합니다. | 동일 AOF 디렉터리로 새 Redis 컨테이너를 기동한 뒤 `WAITING` 토큰과 순번을 복구했습니다. |
| 대기 상태 유실 | 확인할 수 없는 기존 토큰은 거부하고 사용자가 새 토큰으로 재등록합니다. | 전체 대기 key 유실을 주입한 뒤 기존 토큰 거부와 새 토큰 발급을 확인했습니다. |
| 대기 항목 TTL 만료 | 개별 TTL을 지원하지 않는 ZSET의 오래된 member를 순번 집계와 입장 승격 전에 score 기준으로 제거합니다. | 만료된 member가 backlog와 활성 회차 집계에서 제거되는 것을 확인했습니다. |
| 프로세스 재기동·메트릭 누락 | 자동 제어는 `ENFORCED`로 시작하고, 필수 메트릭이 완전하지 않으면 해제하지 않습니다. | 시작 직후 fail-closed와 메트릭 누락 시 활성화 유지·재활성화를 확인했습니다. |
| `ENFORCED -> OFF` | 신규·기존 요청을 즉시 우회시키고 남은 Redis key는 TTL로 정리합니다. | 기존 대기자가 Redis 조회 없이 `BYPASSED` 되는 것을 확인했습니다. |

관리자 화면은 DB에 저장된 `설정 모드`와 자동 제어가 반영된 `실효 모드`를 분리해 표시합니다. `SHADOW`에서 임계값으로 자동 활성화되면 설정값을 바꾸지 않고 `실효 모드=ENFORCED`와 자동 활성화 여부를 주기적으로 갱신합니다.

Redis 중단 응답은 평균 약 `4.08s`가 걸렸습니다. 정책상 우회는 막았지만 Redis command timeout만큼 요청 thread가 대기하므로, timeout 값은 장애 부하 테스트를 기준으로 후속 조정합니다.

## 등록 처리량 기준선

Redis와 backend를 기동한 로컬 환경에서 회차 하나에 대기 등록을 집중시켰습니다. 각 요청은 Lua의 `ZADD`와 `ZRANK`를 함께 실행합니다.

| 등록 수 | 성공 | 처리량 | 평균 | p95 | 최대 |
| ---: | ---: | ---: | ---: | ---: | ---: |
| `1,000` | `1,000` | `558.93 req/s` | `297.03ms` | `1.04s` | `1.45s` |
| `10,000` | `10,000` | `1,936.14 req/s` | `223.27ms` | `287.22ms` | `657.15ms` |

두 실험은 VU와 warm-up 조건이 달라 서로의 우열을 비교하는 값이 아니라, 오류 없이 backlog를 생성할 수 있는 기준선입니다. 현재 입장 허용값 `20명 / 3초`는 약 `6.67명/s`이므로 `10,000`명을 모두 처리하면 이론상 약 `25분`이 걸려 대기 TTL `15분`보다 깁니다. 보호할 backend 처리량을 측정하지 않은 상태에서 배치를 임의로 늘리지 않고, 입장률·TTL 불변식과 자동 활성화 임계값은 후속 이슈에서 함께 조정합니다.

## 자동 활성화 교정 결과

`5 -> 10 -> 15 -> 20 -> 30 -> 5 req/s`로 도착률을 단계적으로 높였다가 낮추는 전체 결제 여정으로 측정했습니다.

| 항목 | 결과 |
| --- | ---: |
| 전체 여정 | `999` |
| 결제 완료 | `999` |
| 예상 밖 오류 | `0` |
| 자동 활성화 시점 | 약 `16.98 req/s` |
| 전체 여정 p95 | `13.25s` |
| 대기시간 p95 | `13.01s` |
| 최대 대기 인원 | `178` |
| Hikari pending 최대 | `0` |
| Tomcat busy 최대 | `8` |
| 예약 동시 실행 최대 | `6` |

첫 실행에서는 자동 전환 직전에 `SHADOW`로 통과한 3건이 예약 시점의 `ENFORCED`에서 `428`로 거부됐습니다. 사용자·회차별 bypass permit을 추가한 뒤 같은 조건을 재실행해 `999/999` 결제 완료와 예상 밖 오류 `0`을 확인했습니다.

현재 `15 req/s`는 서버의 최대 처리량이 아니라 이 테스트의 입장률과 자원 제한에서 backlog가 발생하기 전에 대기열을 켜기 위한 초기 교정값입니다. 입장 batch와 인프라가 달라지면 동일한 단계 측정으로 다시 조정합니다.

## 후속 고도화

1. 다중 인스턴스 전환 시 자동 전환 판단 주체와 상태 공유 방식을 결정합니다.
2. 운영 관측값을 기준으로 임계값과 입장 batch를 함께 재교정합니다.
3. 대기 TTL 안에 backlog tail이 모두 입장할 수 있는지 지속 검증합니다.
