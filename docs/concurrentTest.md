# 동시성 테스트 계획서

## 목적

현재 예매 시스템에서 겹치는 좌석 묶음에 대해 동시에 예약 요청이 들어올 때  
다음 항목을 확인한다.

- 중복 예약이 발생하는지
- 실패율이 어느 정도인지
- 성공 요청이 몇 건인지
- 좌석 상태가 일관되게 유지되는지

---

## 테스트 범위

이번 테스트는 **ReservationGroup 기준 좌석 예약 동시성 문제만** 확인한다.

확인 대상:
- `createReservation()`
- 겹치는 `seatIds`에 대한 동시 요청
- 좌석 id 정렬 후 `PESSIMISTIC_WRITE` 조회

제외 대상:
- 결제 승인
- 중복 결제
- 결제 실패
- 환불/취소 정책

---

## 테스트 시나리오

### 시나리오 1
- 겹치는 `seatIds`
- 여러 명의 사용자
- 동시에 예약 요청

예시:
- 좌석 묶음 `[1, 2]`, `[2, 3]`
- 사용자 20명
- 동시에 `createReservation()` 호출

---

## 확인 항목

### 1. 성공 건수
- 총 몇 건의 예약이 성공했는지 확인

### 2. 실패 건수
- 총 몇 건이 실패했는지 확인

### 3. 실패율
- 실패 건수 / 전체 요청 수

### 4. 중복 예약 발생 여부
- 동일한 좌석으로 Reservation이 2건 이상 생성되었는지 확인

### 5. 좌석 최종 상태
- 최종적으로 Seat 상태가 어떤 값인지 확인

---

## 기대 결과

정상이라면:
- 1건만 성공
- 나머지는 실패
- 겹치는 좌석에 대한 중복 예약 없음
- 선택 좌석은 전체 성공 또는 전체 실패

문제가 있다면:
- 2건 이상 성공
- 동일 좌석에 대해 여러 Reservation 생성
- 중복 예약 발생

---

## 현재 검증 결과

2026-05-28 기준 `ReservationConcurrencyTest`로 group 기준 동시성 테스트를 실행했다.

검증 시나리오:
- 사용자 10명이 동시에 예약 생성 요청
- 요청 좌석 묶음은 `[A-1, A-2]`, `[A-2, A-3]` 형태로 교차
- 모든 요청은 공통 좌석 `A-2`를 포함하므로 동시에 모두 성공하면 안 된다

검증 결과:
- 성공 요청: 1건
- 실패 요청: 9건
- 저장된 reservation: 성공 group의 좌석 2건
- 저장된 group: 1건
- 최종 HELD 좌석: 2건

결론:
- 겹치는 좌석 묶음 동시 요청에서 하나의 `ReservationGroup`만 전체 성공한다.
- 실패 요청은 부분 reservation을 남기지 않는다.
- `Seat` 비관적 락과 id 정렬 정책이 group 단위 원자성 보장에 유효하게 동작한다.

## 락 순서 정책

다중 좌석 예약에서는 서로 다른 요청이 같은 좌석들을 다른 순서로 잡으려 하면 데드락 가능성이 커진다.

예:
- A 요청: `[1, 2]`
- B 요청: `[2, 1]`

이를 줄이기 위해 `ReservationService`는 요청 seat id를 정렬하고, `SeatRepository.findAllByIdInForUpdate()`도 `order by s.id asc` 기준으로 조회한다.

정책:
- 애플리케이션 레벨에서 `seatIds.distinct().sorted()` 적용
- DB 조회에서도 `order by s.id asc` 적용
- 같은 자원 집합은 같은 순서로 락 획득을 시도한다.
