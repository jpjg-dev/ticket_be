# 백엔드 기능 확장 로드맵

## 문서 목적

이 문서는 TicketLedger 백엔드의 후속 확장 후보와 리팩토링 후보를 정리합니다. 현재 포트폴리오의 핵심 범위는 예약/결제 정합성과 성능 최적화이며, 이 문서는 그 다음 단계에서 무엇을 넓히고 무엇을 다듬을지 보여주기 위한 자료입니다.

## 현재 구현 범위

| 범위 | 현재 상태 |
| --- | --- |
| 좌석 선점 | `ReservationGroup` 기준으로 다중 좌석을 선점합니다. |
| 예약 만료 | 결제하지 않은 선점 좌석을 만료 후 복구합니다. |
| 결제 승인/취소 | 승인은 `CONFIRMING`, 취소는 `CANCELING` 회색지대를 거쳐 PG 호출 전후 트랜잭션을 대칭으로 분리합니다. 취소는 소유자 검증 후에만 수행합니다. |
| 결제 보정 | 오래된 `CONFIRMING`/`CANCELING` 결제를 PG 조회 결과 기준으로 승인/실패/환불/취소 확정 처리합니다. 결정 규칙은 순수 함수 정책 객체로 분리했습니다. |
| 운영 관측 | 회색지대 보정 결과, PG 호출 실패, 회색지대 backlog를 Micrometer 지표로 노출합니다. |
| 정합성 검증 | 동시성 테스트와 E2E 부하 테스트 후 중복 좌석/부분 성공/상태 불일치 `0`을 확인했습니다. |

## 리팩토링 후보

| 후보 | 이유 |
| --- | --- |
| 회색지대 조회 인덱스 추가 | 보정 스케줄러가 한 주기에 `status` 기준 조회를 여러 번 수행합니다. 결제 볼륨이 커지면 `payments(status, confirming_at)` / `(status, canceling_at)` 인덱스를 검토합니다. |
| 보정 재시도·알림 체계 | retry 카운터, 백오프, dead-letter, `HOLD_MANUAL` 알림 임계는 Kafka와 분산 락·분산 트랜잭션을 도입하고 서비스/DB를 분해하는 시점에 함께 설계합니다. |

> 완료됨: 보정 환불 호출 트랜잭션 밖 분리, 결제 보정 정책 객체 분리, 결제 취소 `CANCELING` 중간 상태 도입, 운영 관측 지표 보강, 관측성 패키지 중립화, PG·JWT 출력 포트 분리, 공연/좌석 조회 책임 분리, ArchUnit 계층 규칙 추가.

## 기능 확장 후보

### 모바일 티켓 및 입장 처리

- [ ] 결제 완료 후 모바일 티켓 발급 구조를 추가합니다.
- [ ] `Reservation 1건 = 좌석 1개 = Ticket 1개` 기준으로 `Ticket` 엔티티를 추가하는 방식을 검토합니다.
- [ ] 결제 승인 성공 시 group 안의 각 reservation에 대해 Ticket을 생성합니다.
- [ ] 기존 예약/결제 도메인을 다시 설계하기보다 Ticket 도메인을 추가로 붙이는 확장으로 봅니다.

### 티켓 상태 전이 정책

후보 상태:

```text
ISSUED -> USED
ISSUED -> CANCELED
ISSUED -> EXPIRED
```

설계 기준:

- 예약/결제 상태는 "좌석을 구매했는가"를 나타냅니다.
- 티켓 상태는 "현장에서 입장했는가"를 나타냅니다.
- 결제 취소 시 연결된 Ticket은 `CANCELED` 처리되어야 합니다.

### 티켓 사용 처리 API

- [ ] 관리자 또는 검표 담당자가 모바일 티켓의 QR/티켓 번호를 확인해 `USED` 처리하는 API를 검토합니다.
- [ ] 이미 `USED` 처리된 티켓은 다시 사용할 수 없어야 합니다.
- [ ] 입장 처리도 상태 전이이므로 동시 검표 요청에 대한 정합성 검증이 필요합니다.

## 설계 기준

| 기준 | 설명 |
| --- | --- |
| 기존 예약/결제 흐름 유지 | 핵심 포트폴리오 범위인 예약/결제 정합성을 흔들지 않습니다. |
| 새 도메인 추가 중심 | Ticket 도메인을 추가하고 기존 모델 변경은 최소화합니다. |
| 상태 전이 우선 | 티켓도 `ISSUED`, `USED`, `CANCELED` 같은 상태 전이로 관리합니다. |
| 동시성 검증 포함 | 중복 입장 처리 같은 현장 운영 문제도 테스트 대상으로 봅니다. |
