# Backend Feature Roadmap

TicketLedger 백엔드의 추후 기능 확장 후보를 정리한다.

이 문서는 기존 동작을 유지한 채 내부 구조를 개선하는 리팩토링 목록이 아니라, 아직 구현하지 않은 도메인 기능을 추가하기 위한 로드맵이다.

## 1. 모바일 티켓 및 입장 처리

- [ ] 결제 완료 후 모바일 티켓 발급 구조 추가
  - 현재 시스템은 예약/결제 상태만 관리하고, 실제 공연장 입장 여부는 별도로 관리하지 않는다.
  - 실물 티켓 발권 대신 모바일 티켓 형태로 발급하는 방향을 검토한다.
  - `Reservation 1건 = 좌석 1개 = Ticket 1개` 기준으로 Ticket 엔티티를 추가하는 방식이 자연스럽다.
  - 결제 승인 성공 시 group 안의 각 reservation에 대해 Ticket을 생성한다.
  - 기존 예약/결제 도메인을 다시 설계하기보다, Ticket 도메인을 추가로 붙이는 확장으로 본다.
- [ ] 티켓 상태 전이 정책 정의
  - 후보 상태: `ISSUED`, `USED`, `CANCELED`, `EXPIRED`.
  - 예약/결제 상태는 "좌석을 구매했는가"를 나타내고, 티켓 상태는 "현장에서 입장했는가"를 나타낸다.
  - 결제 취소 시 연결된 Ticket은 `CANCELED` 처리되어야 한다.
- [ ] 티켓 사용 처리 API 추가
  - 관리자 또는 검표 담당자가 모바일 티켓의 QR/티켓 번호를 확인해 `USED` 처리하는 API를 검토한다.
  - 이미 `USED` 처리된 티켓을 다시 사용할 수 없도록 중복 사용 방어가 필요하다.
  - 입장 처리도 상태 전이이므로 동시 검표 요청에 대한 정합성 검증이 필요하다.

## 설계 기준

- 기능 확장은 기존 예약/결제 흐름을 깨지 않고 새 도메인 모델을 추가하는 방식으로 진행한다.
- 기존 코드 변경은 결제 승인 후 Ticket 생성 이벤트나 호출 지점을 추가하는 수준으로 제한한다.
- 파라미터 추가가 필요할 수는 있지만, 예약/결제 핵심 모델을 다시 갈아엎는 방향은 피한다.
