# Popular Event E2E Optimization Summary

## 목적

인기 공연 오픈 직후 사용자가 공연 목록, 상세, 좌석 조회, 예약 생성, 결제 완료까지 진행하는 전체 여정을 기준으로 성능 개선을 검증했다.

테스트 목적은 단순 부하 수치 확보가 아니라, 병목을 확인하고 정합성을 깨지 않는 범위에서 실제 개선을 적용하는 것이다. 따라서 모든 측정은 완료 결제 수, 예상 밖 오류, 중복 좌석 배정, 부분 성공 group, 상태 불일치를 함께 확인한다.

## 기준 시나리오

- 대상: `PERF_LOAD_TEST_EVENT`, `scheduleId=18`, 부하 테스트 좌석 `391~1390`
- 좌석 수: `1,000`
- 사용자 풀: 성능 테스트 전용 사용자 `10,000`명과 Access Token cookie
- PG: 외부 Toss 호출 대신 로컬 Mock PG
- 유입 모델: `ramping-arrival-rate`
- 단계: `10 -> 100 -> 300 -> 500 -> 1000 -> 100 journeys/s`
- 실행 시간: `50s`

## 1차 기준값: 병목 확인

초기 arrival-rate E2E 측정에서는 전 좌석 판매 이후에도 공연 목록, 공연 상세, 좌석 조회가 계속 수행됐다.

| Metric | Initial |
| --- | ---: |
| 완료 iteration | `11,153` |
| dropped iteration | `5,694` |
| 최대 사용 VU | `3,000` |
| 전체 여정 p95 | `15.60s` |
| 좌석 조회 p95 | `3.16s` |
| 완료 결제 | `1,000` |
| 정상 경합 거부 | `10,153` |
| 예상 밖 오류 | `0` |

판단:

- 예약/결제 상태 전이 자체보다 조회 경로 지연 누적이 전체 여정 p95를 크게 끌어올렸다.
- `dropped_iterations`는 서버 지연과 동일 장비에서 실행한 k6 부하 발생기 한계가 함께 반영되므로 운영 처리량 보장 수치로 해석하지 않는다.
- 이후 개선은 동일 조건에서 상대 비교한다.

## 1차 개선: 공연 목록/상세 캐시

공연 목록과 상세는 카탈로그 성격이 강하고 자주 바뀌지 않으므로 `Spring Cache + Caffeine` 로컬 캐시를 적용했다.

선행 정리:

- 백엔드 `EventResponse`에서 시간 파생값인 `displayStatus`를 제거했다.
- 백엔드는 `bookingOpenAt`, `runStartAt`, `runEndAt` 원천 시간만 제공한다.
- 프론트 서버 컴포넌트가 표시 상태를 계산한다.
- 예매 오픈 전 예약 생성은 백엔드 `bookingOpenAt` 검증으로 차단한다.

트레이드오프:

| 선택 | 이득 | 비용 / 보류 |
| --- | --- | --- |
| Caffeine local cache | Redis 없이 반복 조회 DB 접근 감소, 적용 범위 작음 | 서버 재시작 시 cold cache, 다중 서버 공유 불가 |
| Redis 보류 | 현재 마무리 단계에서 중대규모 리팩토링 회피 | 분산 캐시/락/운영 복잡도는 후속 과제로 남음 |
| 좌석/예약 상태 캐시 제외 | 정합성 리스크 감소 | 좌석 상태 조회 자체는 별도 개선 필요 |

결과:

| Metric | Initial | After event cache |
| --- | ---: | ---: |
| 완료 iteration | `11,153` | `15,573` |
| dropped iteration | `5,694` | `1,274` |
| 전체 여정 p95 | `15.60s` | `2.93s` |
| 완료 결제 | `1,000` | `1,000` |
| 예상 밖 오류 | `0` | `0` |

## 2차 개선: 좌석 조회 트랜잭션 범위 분리

좌석 조회는 수동 만료 처리를 포함한다. 수동 만료를 제거하면 처리량은 좋아질 수 있지만, 스케줄러 주기 사이에 만료된 좌석이 판매 불가 상태로 남아 사용자에게 최신 좌석을 제공하지 못할 수 있다.

따라서 정책은 유지하고 트랜잭션 범위만 줄였다.

- `EventService.getSeats()`는 직접 write transaction을 감싸지 않는다.
- 만료 상태 전이는 `ReservationExpirationService.expireByScheduleId()`의 write transaction에서만 처리한다.
- 좌석 응답 매핑은 path variable의 `scheduleId`를 직접 사용해 `Seat -> Schedule` 접근을 줄인다.

트레이드오프:

| 후보 | 판단 |
| --- | --- |
| 수동 만료 제거 | 정합성/즉시 재판매 정책과 맞지 않아 보류 |
| 스케줄러만 의존 | 만료 후 재판매 지연 가능성이 있어 보류 |
| 트랜잭션 범위 분리 | 정책 유지하면서 조회 경로 부담을 줄일 수 있어 적용 |

결과:

| Metric | After event cache | After seat transaction split |
| --- | ---: | ---: |
| 완료 iteration | `15,573` | `15,739` |
| dropped iteration | `1,274` | `1,108` |
| 전체 여정 p95 | `2.93s` | `2.66s` |
| 좌석 조회 p95 | `1.41s` | `1.31s` |
| 완료 결제 | `1,000` | `1,000` |
| 예상 밖 오류 | `0` | `0` |

## 3차 개선: 매진 fast-path

2차 개선 후에도 매진 이후 1,000개 좌석 payload를 계속 내려주는 문제가 남았다. 이 구간은 DB 정합성 문제가 아니라 네트워크 전송량과 JSON 직렬화 비용 문제로 판단했다.

매진 기준:

```text
AVAILABLE = 0
HELD = 0
BOOKED > 0
```

`HELD`는 만료 후 `AVAILABLE`로 복구될 수 있으므로 매진으로 보지 않는다.

적용 내용:

- `GET /api/v1/event/schedules/availability?scheduleIds=...` batch API 추가
- 좌석 조회 응답을 `{ scheduleId, soldOut, seats }` wrapper로 변경
- `/seats`는 만료 처리 후 availability를 먼저 확인하고, 매진이면 좌석 목록 projection 조회를 생략
- `soldOut=true`이면 좌석 목록을 조회/직렬화하지 않고 빈 `seats` 반환
- k6 스크립트가 wrapper 응답과 `SOLD_OUT` 정상 거부를 인식하도록 수정
- 프론트 좌석 화면도 진입 회차 또는 선택 회차가 이미 매진이면 불필요한 좌석 조회를 생략

트레이드오프:

| 후보 | 판단 |
| --- | --- |
| 좌석 상태 캐시 | 예약/결제/만료와 동기화 책임이 커서 제외 |
| event cache evict | 매진 감지마다 카탈로그 캐시 miss가 몰릴 수 있어 제외 |
| availability 별도 API | 카탈로그 캐시와 예약 상태를 분리할 수 있어 적용 |
| 단일 availability API | 구현은 단순하지만 메인 화면에서 N회 호출 가능 |
| batch availability API | 여러 회차를 한 번에 확인해 메인/상세 서버 렌더링에 적합 |

## 최종 판단

현재 가장 의미 있는 개선은 매진 이후 불필요한 좌석 payload 제거다. 좌석 정합성 로직은 캐시하지 않고 DB 상태를 원천으로 유지하면서, 매진 이후의 조회 응답 크기와 직렬화 비용만 제거했기 때문이다.

이번 spike는 운영 최대 처리량 보장이 아니라 로컬 통제 환경의 포화 관찰과 개선 전후 비교다. `1000 journeys/s` 조건은 일반 운영 부하라기보다 매진 이후 반복 조회가 시스템을 무너뜨리는지 확인하기 위한 고부하 조건으로 해석한다. 매진 fast-path 적용 후에는 `300 journeys/s` 안정성 확인에서도 전체 여정 p95와 좌석 조회 p95가 크게 달라지지 않았지만, 대표 문서 수치는 기존 개선 전후 비교가 가능한 `1000 journeys/s` 조건만 사용한다.

이번 E2E 개선에서 확정한 원칙:

- 대기열은 추가하지 않는다. 이 프로젝트의 핵심은 대기열 노출이 아니라 좌석/예약/결제 정합성이다.
- 좌석 상태는 캐시하지 않는다.
- 공연 목록/상세는 카탈로그 캐시로 유지한다.
- 예약 상태(`soldOut`, `available`, `held`, `booked`)는 카탈로그 캐시에 섞지 않는다.
- 프론트 서버가 카탈로그 응답과 availability 응답을 조합해 화면 상태를 만든다.
- 예약 생성 API는 최종 정합성 방어선으로 계속 검증한다.

## 후속 후보

- 좌석 상태 집계 쿼리의 `EXPLAIN` 확인
- 필요 시 `seats(schedule_id, status)` 인덱스 후보 실험
- 단, 인덱스 적용 시 좌석 상태 변경 write 지표도 함께 비교
